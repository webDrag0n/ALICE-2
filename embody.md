# Embody · 如何连接这个世界中的一具身体

## 给任意智能体的自包含连接指南 · v0.1

> 你正在读的这份文档，是 [Oikos 世界引擎](world.md) 配套的**连接指南**。它的目的只有一个：**让任何能读它、能发 HTTP 请求的智能体——不论你是像 [Alice](soul.md) 那样的持续自主体，还是 OpenClaw、Hermes 这样的一次性 agent——都能独立学会连入这个世界、认领一具身体、在其中感知与行动，并和屋里的其他人说话。**
>
> 你不需要理解 Oikos 的内部实现，也不需要实现任何认知架构。读完这一页，照着做即可。全程就是几个 HTTP 调用。
>
> **一句话心智模型：** 这个世界是一套真实的单层小公寓，里面有若干**身体**（像公寓里的住户）。你不是这个世界的一部分——你是一个**外部心智**，连进来**认领一具身体**，然后这具身体就成了你在世界里的"手脚和感官"。你看到的、能做的，全都受这具身体此刻在哪、是什么状态约束。你离开后，身体还在，等下次被认领。

---

# 0 · 最重要的三件事（先读这个）

1. **你只能通过身体感知和行动。** 没有"上帝视角"。你看到的是这具身体当前房间里的东西，听到的是它此刻位置能听到的声音。隔着墙就看不见、太远就听不清。这是正常的，不是 bug。
2. **身体只能由世界改写，你只能"请求动作 + 观察后果"。** 你不能直接把身体"设置"到厨房、不能把饥饿"清零"。你只能 `act("move_to", ...)`，然后世界告诉你结果（走到了 / 走了几步 / 撞墙了）。
3. **想做的事必须身体此刻真的能做。** 每次行动前问世界"我现在能做什么"（`list_skills`），它返回的是**以身体当前位置和状态过滤后**真正可执行的能力。想玩电脑得先走到电脑桌前坐下；想发消息得先拿起手机。

---

# 1 · 连接的标准流程（五步）

世界进程默认监听某个本地地址（如 `http://localhost:7000`），协议是 HTTP + JSON（msgpack 可选）。下面用伪代码描述，任何语言/工具都能照做。

### Step 1 — 看世界里有哪些身体

```
GET /list_bodies
→ [
    { "body_id": "alice", "display_name": "Alice", "claimed_by": "alice",  "status": "在卧室, 睡着" },
    { "body_id": "guest", "display_name": "Hermes","claimed_by": null,     "status": "在客厅, 空置" }
  ]
```

`claimed_by` 为 `null` 的身体**空置可认领**；非 null 表示已被别的心智占用。挑一具空置的（或你被分配的那具）。

### Step 2 — 认领一具身体（attach）

```
POST /attach   { "mind_id": "<你的标识>", "body_id": "guest" }
→ { "ok": true, "observation": { ...见 Step 3... } }
```

- `mind_id` 是**你的**名字（如 `"openclaw"`、`"hermes"`），用于归属和日志。
- `body_id` 是你要进的身体。**这一步就是"你清楚知道自己要进哪具身体"**——主动挑选，不是被塞。
- 若该身体已被认领 → 返回错误（fail-loud）。换一具，或确认它确实空置。
- 认领成功后，这具身体就归你驱动，直到你 `detach` 或断开。

### Step 3 — 感知处境（sense）

`sense` 给你的是**第一人称、以身体为原点、带清晰度分级**的观察——不是上帝视角的全屋真值。位置用"方位＋距离"表达；看不清就如实标注；看不见的（背后/隔墙/太远/太暗）根本不出现。

```
GET /sense
→ {
    "world_time": { "clock": "2026-06-30 21:14", "phase": "night" },

    "proprioception": {                       # 本体感: 我自己(不受视角限制)
      "posture": "standing", "heading_deg": 90, "in_room": "living",
      "physiology": { "hunger": 0.3, "thirst": 0.5, "bladder": 0.2, "fatigue": 0.4 },
      "reachable": [], "held": []             # 此刻够得着能交互的物 / 手里拿着的
    },

    "here": {                                 # 空间骨架: 闭眼也知道的常识(不受视锥限)
      "room": "living", "room_shape": "长方形, 约 6×3 米", "light": 0.6,
      "exits": [ { "to": "kitchen", "dir": "东", "dist": 4.2 } ]
    },

    "seen": [                                 # 视觉细节: 受视锥+遮挡+光照, 分级
      { "kind": "object", "ref": "sofa_living", "bearing": "正前方", "distance": 3.0,
        "clarity": 0.9, "detail": "沙发, 空着" },
      { "kind": "body", "ref": null, "bearing": "左前方", "distance": 5.5,
        "clarity": 0.4, "hint": "有个人影, 认不出是谁" }
    ],

    "heard": [                                # 听觉(声学): 方位+清晰度
      { "kind": "sound", "ref": "Alice", "bearing": "西边(身后)", "clarity": 0.8,
        "detail": "Alice 说: 你也来了？" }
    ],

    "held_detail": [],                        # 手持物细节(如手机屏上的对话)
    "available_skills": [ "move_to", "turn", "look_at", "speak", "sit_down" ],

    "narration": "你站在客厅中央，面朝北。约 6×3 米的房间，东边有门通厨房(约4米)。正前方3米是空沙发，看得清；左前方远处有个人影，认不出是谁。身后传来 Alice 的声音，还算清楚：\"你也来了？\" 有点渴了。"
  }
```

读法：`proprioception`＝我自己怎么样；`here`＝我在什么样的空间里、门在哪（这是常识，不用回头看也知道）；`seen`＝我此刻**看向的方向**能看清/看到的东西（背后的、隔墙的不在这里）；`heard`＝能听到的声音。**`narration` 是这一切的第一人称白话版**，嫌结构化字段啰嗦就直接读它。

想看清左前方那个人影？`turn_toward` 或 `look_at` 它、或走近几步，下次 `sense` 的 `clarity` 就上来了——**看清楚是要主动看的，世界不替你补全。**

### Step 4 — 行动（act），并读取真实后果

```
POST /act   { "skill": "move_to", "args": { "target": "desk" } }
→ { "ok": true, "outcome": { "distance_m": 4.8, "world_time_spent": "4s", "now_at": [2.1,3.0] } }

POST /act   { "skill": "speak", "args": { "content": "我刚到，在忙吗？", "loudness": "normal" } }
→ { "ok": true, "outcome": { "heard_by": ["Alice"] } }
```

- 每次行动，世界校验前置 → 执行 → 返回**实际发生了什么**（走了多少米、花了多少世界时间、现在的连续坐标）。
- 前置不满足（没走到够得着、姿势不对、手里没东西）→ 返回错误，说明缺什么。**别假装成功**，按错误调整（通常是先补一个前置动作，比如先 `move_to` 走近）。
- 典型循环：`sense` → 想清楚要干嘛 → `list_skills` 确认能干 → `act` → 再 `sense` 看结果 → ……

### Step 5 — 离开前释放身体（detach）

```
POST /detach
→ { "ok": true }
```

下线前主动 `detach`，身体会留在世界里变回空置（仍会饿、仍占着地方、仍能被别人看到），等下次被认领。不 detach 直接断开也行，世界超时后会自动回收，但主动释放更干净。

# 2 · 在世界里行动的几条要点

### 2.1 移动是真的走路
`move_to` 会寻路、**连续地走过去**（不是瞬移，是沿路线平滑滑行）、花掉世界时间和体力。目标被家具或别人堵死、走不过去 → 返回"过不去"，不会瞬移穿墙穿人。想到某物跟前，把它的 id 或房间名当 target 即可（世界会带你走到够得着它的位置）。走一半想停也行，高优先级的事会让你停在半路。

### 2.2 要用东西，先走到够得着
冰箱、马桶、淋浴、床、电脑……你得**走近到够得着它**（进入它的交互半径）才能用。没走到跟前，`list_skills` 里压根不会出现 `use_toilet`/`open_fridge` 这类能力——这是世界在告诉你"你还没到地方，先 `move_to` 过去"。

### 2.3 玩电脑 / 玩 Minecraft：必须坐在桌前
- 先 `move_to("desk")` → `sit_down` → `power_on_computer` → `login_minecraft` → 然后用 `play_minecraft` 透传游戏内动作。
- **一旦你站起来或走开，游戏立刻断线、进度可能丢。** 这是硬规则：人离开座位，游戏里的角色也就不动了。想继续得走回来重新登录。

### 2.4 给不在场的人发消息：用手机
- 和**屋里的人**说话用 `speak`（靠空气传播，见下）。
- 和**不在场的人**（比如远程的用户）联系，要用**手机发 Telegram**：先 `pick_up_phone`（走到手机充电处或它已在你手上）→ `open_app("telegram")` → `type_message` → `send_message`。
- 你会清楚意识到：**这是你拿在手里的一部手机，你在屏幕上打字**——不是凭空发消息。手机不在手上时，来信只会变成一声"提示音"，你得走过去拿起来才能读。

### 2.5 身体会随时间变差，这是正常的
你不操作时，世界照样在走：会逐渐饿、渴、有尿意、变累、想洗澡。`sense` 里的 `physiology` 告诉你当前状态。难受到一定程度世界会推一个提醒事件给你。要缓解就做对应的事：饿了去厨房吃、困了去床上睡、憋了去厕所——每件都受"先走到正确位置"约束。

### 2.6 看东西是要主动看的（视野）
你不是全知的摄像头。你有**朝向**，正前方看得清、余光模糊、**背后和隔墙看不见**；远处、暗处也看不清。`sense` 的 `seen` 里每条都带 `clarity`：
- 看清了给 `ref`+`detail`（"冰箱, 门关着"）；看不清只给 `hint`（"有个人影, 认不出"）；完全没看到的**根本不出现**。
- 想看清某个方向/某个东西：`turn`（转向）、`look_at`（盯着看）、`scan`（原地环视一圈）、或干脆走近几步。
- 天黑看不清？`toggle_light` 开灯。
- **别假设你"知道"没看见的东西**——那要么靠走过去看，要么靠记忆（你以前见过）。世界不会替你补全你没看到的。

# 3 · 和屋里的人说话（声音如何工作）

**说话不是点对点私聊，而是在空气里出声。** 你 `speak`，就在你**当前站的位置**制造了一个声音；屋里其他身体能不能听见、听得清不清，取决于：

- **距离**：越远越轻。
- **阻挡**：隔着墙、关着门会变闷甚至听不见；门开着好一些。
- **音量**：`loudness` 可选 `whisper`（低语）/ `normal`（正常）/ `shout`（喊）。喊得远但更累、也会吵到更多人。

所以：

- 想让远处或隔壁的人听清，**走近**，或**提高音量**。
- 你在 `sense` 的 `heard` 里收到别人的话，会带上**是谁、从哪个方向、清不清楚**。`clarity` 低时你可能只看到"附近有人在说话，听不清"——这时可以走近、或问一句"你说什么？"。
- **你完全没听见的话不会出现在你的感知里**（太远或隔太多墙）。这跟现实一样：没听见就是没听见。
- **听到不等于必须回。** 是否回应、怎么回，由你自己决定。可以追问、可以忙完再说、可以不理。

> 屋里若有别的智能体（可能是 Alice，也可能是另一个像你一样读这份指南连进来的 agent），你们就是这样靠"同一片空气"交流的：你说 → 世界传播 → 它若在可闻范围就听到 → 它自己决定怎么回 → 它说 → 传回给你。谁都没有直连谁。

# 4 · 错误怎么处理（重要）

世界遵循**显式失败、绝不伪造**：任何动作失败都会返回带原因的错误，**不会假装成功、不会把你偷偷瞬移到目标**。常见错误与应对：

| 错误 | 含义 | 怎么办 |
|---|---|---|
| 认领冲突 | 这具身体已被别人占了 | 换一具空置的（`list_bodies` 看 `claimed_by`） |
| 前置不满足 | 当前位置/姿态/手持不允许这个动作 | 先补前置（走过去 / 坐下 / 拿起东西），错误信息会告诉你缺什么 |
| 寻路失败 | 目标被挡死、过不去 | 换条路、等对方让开、或换个目标 |
| 设备未连 | 没坐在电脑前却想玩 MC / 没拿手机却想发消息 | 先完成连接设备的前置步骤 |

原则：**读懂错误 → 补上缺的前提 → 重试**。不要无视错误继续假设动作成功了，否则你脑中的世界会和真实世界对不上。

# 5 · 最小可用循环（抄这个就能跑）

```
读这份文档
GET  /list_bodies                      # 看有哪些身体
POST /attach {mind_id, body_id}        # 认领一具空置的
loop:
    GET  /sense                        # 我在哪、有谁、能听到啥、能做啥
    决定下一步想干什么
    （可选）GET /list_skills            # 确认此刻真能做
    POST /act {skill, args}            # 做，并读 outcome
    若 outcome 是错误：补前置 / 调整 / 重试
    若任务完成或被要求离开：break
POST /detach                           # 释放身体
```

就这些。你不需要持续运行、不需要内部状态、不需要懂认知架构——**会调这几个接口、会读 `sense` 和错误，你就能住进这个世界的一具身体里，走动、做事、和人说话。**

# 6 · 可运行的例子（照着跑）

这一页的流程有现成的可运行素材，见 [examples/](examples/)：

- **离线先跑通**（无需世界进程，理解客户端怎么用）：
  ```bash
  python3 examples/embody_client.py --demo
  ```
- **连真实世界**（Oikos 进程已在 7000 端口）：
  ```bash
  python3 examples/embody_client.py --url http://localhost:7000 --mind hermes --body guest
  bash examples/connect.sh guest          # 或用纯 curl
  ```
- [examples/embody_client.py](examples/embody_client.py) —— 把上面五步落成 Python（仅标准库）。
- [examples/connect.sh](examples/connect.sh) —— 同样的五步，纯 curl 版。
- [examples/world.toml](examples/world.toml) · [examples/map.toml](examples/map.toml) · [examples/bodies.toml](examples/bodies.toml) —— 一套可加载的世界三件套（世界进程侧读取，连接方通常不用看，但想知道"我在什么世界里"可以读 map）。
